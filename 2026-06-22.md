## Notes


We are currently stuck at 2.01 million calls a second. 
We might have **FPU Overhead**: Because your SAMD51 processor features a dedicated hardware Floating Point Unit (FPU), the cost of converting int8 weights back to float during the forward pass is extremely low.

The bottleneck is probably **the Flash Memory bandwidth** (the CPU is stalled, waiting). The Wio Terminal has an ARM Cortex-M4F CPU running at 120 MHz. It has a hardware Floating Point Unit (FPU) that can execute **a multiply-and-add in 1 to 2 clock cycles**. 

However, the model weights are stored in **Flash Memory** (`PROGMEM`), not RAM.
Every time the  code executes `pgm_read_byte(&fc1_w[...])`, the CPU has to pause and wait for the Flash memory controller to fetch that byte. This Flash read latency takes **several clock cycles**. Furthermore, your `for` loops have overhead (incrementing `j`, checking `j < 768`, branching).


Here are some ideas to maximize the number of calls per second, based on priority:

- Unroll the Loop: Instead of calling forward() once per loop, call it 10 or 100 times per iteration, and count the new total number.

- Let's make sure the bottleneck is not the serial/display printing loop or the way the loop is timed.

- comment out Serial.print() and TFT_eSPI update functions to find the "true" inference speed of the neural network?

- Check Memory Latency: Since your weights are stored in PROGMEM, the inference speed is ultimately limited by the memory bus speed of the Wio Terminal when fetching these parameters. You can only exceed the current rate if you move to a model that fits entirely within the processor's internal cache or if you optimize the weight access pattern to be more cache-friendly.
